Emergent

Emergent 是 AI app generation 赛道里增长最快、但公开信息最少的年轻公司之一。

1一句话结论

Emergent 是 AI app generation 赛道里增长最快、但公开信息最少的年轻公司之一。

它与 Lovable/Replit/Bolt 同属 prompt-to-app/vibe coding，但定位更激进: 让非技术用户通过自然语言生成 full-stack, production-ready web/mobile apps，并用 E3、Kubernetes real environments、Fable/Opus/Codex 等模型/agent architecture 提高复杂构建和部署成功率；2026 年还推出 Wingman personal AI agent，把 app builder 能力延伸到 WhatsApp/iMessage/Gmail/Calendar/Slack 等个人工作流。

公开报道显示，Emergent 2026 年 1 月 Series B 融资 $70M，投资方包括 Khosla Ventures、SoftBank Vision Fund 2、Prosus、Lightspeed、Together、YC；同月 BI 报道 5M users、ARR 从 $100K 到 $50M/7个月、80% 用户从未写过代码，2 月披露 $100M ARR/8个月，4 月披露 8M+ users 与 3 月 revenue $8.4M。核心投资问题是: Emergent 能否把高速增长和低价 app generation 转成高质量、低 churn、可安全上线的软件创建平台，同时避免被 Lovable/Replit/Bolt 和大模型公司前向整合。

Emergent 的一个显著特征是“年轻但增长曲线极陡”。与 StackBlitz/Bolt 和 Replit 不同，它没有长达七八年的 developer infra 历史；与 Lovable 不同，它公开估值更低、融资规模更小；但它用很短时间拿到 5M-8M users 和 $50M-$100M ARR 线索，并在产品 blog 中持续讨论部署成功率、Kubernetes 环境、Fable 5、E3 orchestrator 等较底层的 engineering choices。

这意味着 Emergent 可能是赛道里最典型的“AI-native app factory”公司: 用强模型和 agent architecture 直接面向非技术用户生成可运行产品。它的 upside 是增长快、资本效率高、估值相对头部低；downside 是公开数据少、质量/安全/留存未验证、产品和组织仍早期。

Emergent 的价格比 Lovable/Replit/Bolt 的高档企业产品更简单: Free -> Standard -> Pro -> Enterprise。个人 Standard 年付 $17/月、Pro 年付 $167/月与 BI 报道一致。Pro 的“1M context window、Ultra thinking、custom AI agents、high-performance computing”说明它把更贵的长上下文/高 reasoning/复杂 agent 作为高 ARPU tier，而不是让低价用户无限消耗。

4.2 Credits 与 extra credits

Emergent 的 Standard 提供 100 credits/month，Pro 提供 750 credits/month，Free 提供 10 credits/month。官方页面没有完全公开每类动作的 credit cost，但公司 blog 关于 Fable 5 的文章提供重要成本线索:

  • Fable 5 在 matched paid builds 上 median project 约 66 credits，对比 Opus 4.8 约 44 credits，约 1.5x。
  • Fable credit rate 为 Opus 的 2x，但因 steps/rounds 更少，完成 build 不一定 2x 成本。
  • Fable first build median active time 16 minutes vs Opus 10 minutes，更多工作前置。
  • Fable production steps 中 23.5% route to specialist sub-agents vs Opus 14.6%。
  • Prompt caching 命中率约 93%-95%，说明平台在控制长上下文成本。

这比多数年轻 app builder 披露得更细。它显示 Emergent 已经把 credit pricing 与底层模型/agent step/runtime 成本绑定，而不是简单收月费。

4.3 Wingman pricing

BI 2026 年 4 月报道称 Wingman 的 pricing starts with a small subscription fee, followed by usage-based pricing。Wingman 是 personal AI agent，不是 app builder。它拥有自己的 phone number/email，可接入 WhatsApp、iMessage、Gmail、Google Calendar、Slack 等，用户像给人类助理发消息一样委派任务。因为这类任务可能涉及多步执行、外部 app 操作、email draft、research、scheduling，usage-based pricing 合理。

4.4 成本结构

Emergent 成本主要包括:

  1. frontier model and reasoning: Fable/Opus/Codex/GPT/Claude 等模型调用，长上下文与高 reasoning 成本高。
  2. agent orchestration: E3 orchestrator、subagents、parallel tool calls、planning/review。
  3. real environments: Kubernetes-based app build/deployment environments，依赖、Docker/runtime、fallback deployer。
  4. hosting/deployment: web/mobile app private hosting、custom domains、IP addresses。
  5. support and repair: 非技术用户生成 production-ready app，需要大量 debugging/support。
  6. personal agent integrations: Gmail/Calendar/Slack/WhatsApp/iMessage 权限、安全、OAuth、audit。
  7. security/red team: Wingman 对权限和 prompt injection/security 极敏感。

5.2 E3: autonomous app building architecture

Emergent 2026 年 6 月发布 E3，官方叙事是 autonomous app building。结合 Fable 5 blog，E3 更像 orchestrator: 决定何时 inspect、何时分解任务、何时让 cheaper executor 完成 routine work、何时用更强 judgment layer 处理长上下文和修复。其核心思想是:

  • E3 不一定每步都用最贵模型；
  • 高级模型用于 planning、review、debugging、architecture choices；
  • cheaper models/subagents 处理执行；
  • 通过 prompt caching、parallel tool calls、budget controls 降低成本；
  • 通过 screenshots、bug detection、deployment feedback 形成闭环。

这和 Cognition Devin 的 autonomous engineer 相似，但 Emergent 更偏“从 prompt 到完整 app”，而不是已有 codebase 上的工程任务。

5.3 Kubernetes real environments

Emergent 2026 年 3 月 blog “Real Environments for AI Agents” 和 2026 年 7 月 “Super Deployer” 很重要。传统部署系统假设写代码的人知道依赖；AI app builder 打破这个假设。Agent 可能写出需要 FFmpeg、Redis、Ollama、CUDA 或特殊 system packages 的 app，但原始部署模板并不知道。Emergent 披露，在构建 Super Deployer 前，约 85% of apps that pushed beyond standard dependency set failed to deploy；原始 pipeline 的静态 manifests 不适合 AI-generated dependency graph。Super Deployer 的思路是保留 happy path，同时为异常依赖构建 fallback。

这说明 Emergent 对“生产可运行”问题有真实工程投入。AI app builder 真正难点不是生成好看的前端，而是依赖、环境、数据库、认证、deployment、runtime errors 和用户没有能力自己修复。

5.4 Fable 5 / cost-quality tradeoff

Emergent 在 Fable 5 blog 中披露多项模型评估:

  • long-horizon eval 中 Fable delivered 45 features vs Opus 4.8 的 41，平均 steps 330 vs 368；
  • uncapped Fable run 成本约 2.5x Opus；
  • 325k task budget 可降低约 17% 成本、11% active build time；
  • early production usage 中 Fable builds new bugs 少约 8%；
  • Fable first build median active time 16 minutes vs Opus 10 minutes；
  • Fable 使用约 80% more credits and 1.7x output tokens；
  • matched paid builds median Fable project 66 credits vs Opus 44 credits；
  • prompt cache 命中约 93%-95%。

这些数据是公司自有口径，但非常适合尽调。它说明 Emergent 正在以 engineering/economics 语言优化 app builder，而不仅靠“模型更强”营销。

5.5 Wingman 与 app builder 的关系

Wingman 看似偏离 app builder，但 Mukund Jha 的解释是: 未来许多 agent 工作，包括做 PPT、研究报告、自动化操作，本质上会通过写代码完成。Emergent 从 coding/app generation 起步，因此有能力让 personal agent 用代码完成更复杂任务。这个 thesis 若成立，Emergent 可从 “build apps” 扩到 “build and execute workflows”。

风险也很明显: personal agent 要处理 email、calendar、messages、Slack、phone identity，权限和 prompt injection 面更大，且直接对上 OpenAI/Google/Microsoft/Anthropic/Apple 的 agent products。
